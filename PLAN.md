Benchmark Intelligence - System Architecture & Implementation Plan

1. Executive Summary

Benchmark Intelligence requires a three-part, seamlessly integrated platform to identify, capture, and distribute construction leads in York.

Internal Dashboard & Scraper: An automated system to monitor the York planning portal for "extension" approvals, bypassing session-cookie limitations, alongside a CRM-style dashboard for staff to manage projects.

Homeowner Portal: A customer-facing, mobile-first data capture tool for homeowners to provide contact details and consent, used via iPad (door-to-door) or QR code.

Builder Network Management: A module to manage the internal network of builders and track the bi-directional assignment of projects to specific builders.

2. Technology Stack & Infrastructure

Frontend Framework: React with Vite (Fast compilation, modern module replacement).

Styling: Tailwind CSS (Global configuration, sustainable utility classes, no inline styles).

Icons: Lucide-React (Clean, minimalist, stroke-based icons. No emojis).

Database & Auth: Firebase / Firestore (NoSQL document database, real-time listeners, User Authentication).

Backend / Scraper: Node.js scripts executed via GitHub Actions with puppeteer and cheerio (Crucial for bypassing CORS and handling the council site's session cookies).

Hosting: * Frontend: GitHub Pages (via GitHub Actions CI/CD).

Backend: Firebase Firestore (Serverless Database), Scraper runs on GitHub Actions.

Map Integration: Mapbox GL JS or Google Maps API (for project geographic visualization).

3. Core System Architecture & Data Models

3.1. Firestore Collections

projects: The core entity scraped from the council site.

Fields: id (Council keyVal), address, description, status (System status: New, Contacted, Assigned, Dead), councilStatus, dateDecided, url, notes, collectionId, coordinates (Lat/Lng).

homeowners: Linked 1:1 with projects.

Fields: projectId, fullName, phone, email, consentGiven (boolean), timestamp.

builders: The network entities.

Fields: companyId (e.g., ABC), companyName, ownerName, phone, email, availability (date range/s), preferences (array of strings/tags).

assignments: Junction table mapping Projects to Builders.

Fields: projectId, builderId, dateAssigned, status (Pending, Accepted, Rejected).

4. Detailed Component Breakdown

Part 1: Internal Dashboard & Scraping Pipeline

The Scraper (Node.js Script via GitHub Actions):

The Challenge: The URL weeklyListResults.do requires a session cookie generated by submitting the search form on search.do.

The Solution: A Node.js script running on GitHub Actions will use a headless puppeteer instance. It will navigate to the search page to establish a session, construct the payload for "Decided in this week", capture the resulting redirect HTML, and load the HTML into cheerio.

Filtering & Deep Scraping: It will filter the table for "extension". For each match not already in Firestore, it will follow the link to applicationDetails.do, extract the full description, address, and applicant details, and push it to Firestore.

Triggers: Scheduled automatically via GitHub Actions Cron every Friday at 5 PM, and manually triggerable via GitHub workflow dispatches.

The Internal Dashboard (React UI):

Authentication: Firebase Email/Password auth for staff. Protected route wrapping the app.

Data Grid: A responsive table/grid displaying synced projects. Includes sorting (by date, status), text search, and filtering.

Detail View: A side-panel or modal showing full project details, a rich-text notes field, and a dropdown for the custom status.

Map View: A toggleable view displaying pins for all active projects based on geocoded addresses.

Collections: Ability to select multiple projects (checkboxes) and assign them to a named collection (e.g., "North York Run - 12th Oct").

Part 2: Homeowner Lead Capture

Access Method: To ensure seamless data-linking, the URL accessed via iPad or QR code will contain the project ID: benchmarkintel.com/capture?id=T99ADNSJIQL00.

UI/UX: A highly polished, mobile-first form with large touch targets. Branding should instil trust (Benchmark Intelligence logo, clean typography).

Functionality: * Reads the id from the URL to display the address at the top (e.g., "Regarding your approved extension at 123 Fake Street").

Collects: Name, Phone, Email.

Explicit checkbox for: "I consent to being contacted by Benchmark Intelligence's vetted network of builders for quotes."

Database Update: On submit, creates a document in the homeowners collection and updates the parent project status to "Contacted/Lead Captured".

Part 3: Builder Management & Lead Distribution

Builder Directory: A clean list view of all network builders showing their unique 3-letter ID and current availability toggle.

Assignment UI: Inside the Project Detail view (Part 1), an "Assign Leads" section. Staff can select multiple builders from a searchable dropdown.

Bi-directional Tracking:

Viewing a Project shows tags of which builders it was sent to.

Viewing a Builder profile shows a history list of all projects they have been assigned.

5. UI/UX & Global Styling Guidelines

Vibecoding Prevention: Strict adherence to a professional, brutalist-leaning modernism. No gradients, no emojis, no excessive box-shadows.

Typography: Inter or Roboto (sans-serif, highly readable).

Color Palette (Managed via Tailwind tailwind.config.js variables):

Primary: Slate/Charcoal #0f172a

Secondary: Subdued Blue #0284c7

Background: Off-white/Gray #f8fafc

Surfaces: White #ffffff with subtle 1px borders (#e2e8f0) instead of heavy shadows.

Status Colors: Green (Approved/Lead), Amber (Pending), Red (Dead) using muted, professional shades.

Responsiveness: Use CSS Grid and Flexbox with Tailwind's breakpoint prefixes (md:, lg:). The dashboard collapses to a mobile-friendly list view; the Homeowner form scales perfectly to fill an iPad screen in portrait or landscape.
